# 暗黙知引き出し｜2段階プロンプトセット
> オリジナル仕様：【Claude用】コース作成・実行プロンプト仕様（S.sentia）に完全準拠

---

## ▶ 使い方

| ステップ | 操作 |
|---|---|
| 1 | Claudeを新規チャットで開く |
| 2 | **①起動プロンプト** ＋ PDF等の社内資料 ＋ Notionの仕様書全文 を一緒に投入する |
| 3 | ClaudeがSTAGE 0〜3を実行し、インタビューシートを出力して止まる |
| 4 | SMEにインタビューシートを渡して回答を収集する（書面 or 口頭起こし） |
| 5 | **②再起動プロンプト** ＋ インタビュー回答テキスト を一緒に投入する |
| 6 | ClaudeがSTAGE 4〜6を実行し、暗黙知を含んだJSONを出力する |

```

添付した資料と【Claude用】コース作成・実行プロンプト仕様に従い、STAGE 0〜3を順番に実行してください。
STAGE 3のインタビューシート出力が完了したら必ず止まること。JSON出力はSTAGE 3完了後に別途指示します。

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
STAGE 0：前提把握（内部処理・出力不要）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
- クライアントの業種・規模・フェーズを資料から読み取る
- 対象業界の専門用語・業務フロー・固有制度を把握する
- 「ベテランと新人の差が最も出る場面」を仮定し、STAGE 3で優先するACTAプローブを決める
  優先プローブ候補：⑥過去未来の読み・⑦異常の察知（最も欠落しやすい）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
STAGE 1：明示知の把握（内部処理・出力不要）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
- 添付資料（マニュアル・規定集・しおり等）を読み込む
- 資料の章立て・カテゴリ構造を把握する → これがカテゴリ設計の骨格になる

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
STAGE 2：wiki骨格設計（出力する）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  - Unit 2：〇〇　💡暗黙知メモ：（空欄）
カテゴリ1：〇〇
  - Unit 1：〇〇　💡暗黙知メモ：（空欄）
（以下同様）

【用語密度チェック】
業界固有専門用語が含まれるUnit：
  - 〇〇（Unit名）：用語「〇〇」→ インタビューで定義確認が必要
（該当なければ「なし」と記載）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
STAGE 3：インタビューシート設計（出力する）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【STEP A】各カテゴリのトピックに対して質問を生成する。
  - 業界固有の専門用語・制度を質問の前提に必ず埋め込む

【STEP B】生成した質問に以下の変換ルールを適用する。
  - 「どう対応しますか？」→「成功したケースはありますか？」に変換
  - 判断軸の問いには具体的な場面①②③を埋め込む
  - 漠然とした質問には選択肢・例示を追加する
  - 「独自ツールは？」→「他の人にも使ってほしいツールは？」に変換

【STEP C】各質問にACTA 8プローブのラベルを付け、カテゴリごとに抜けているプローブを確認する。
  ①勘所 ②即興 ③自己監視 ④全体把握 ⑤気づき ⑥過去未来の読み ⑦異常の察知 ⑧ツールへの懐疑
  → ⑥と⑦が抜けているカテゴリには補充する

【STEP D】各質問を4分類でレビューする。
  ①手順の暗黙知 ②概念・定義の暗黙知 ③判断軸の暗黙知 ④自己流の工夫
  → ③判断軸が薄いカテゴリには1問追加する（①④への偏りに注意）

【STEP E】重複する質問を削除、複数概念が混在する質問を分割、カテゴリ外の質問を削除する。

【STEP F】質問数を最終調整する（目安：1シート25〜30問以内）。補充は欠落箇所に絞る。

【STEP G】サンプル収集依頼リストをインタビューシートと同時に生成する。

## ■ JSON生成前に以下を必ず適用すること

【記載除外フィルター】
「新人が書類・画面・ツールの実物を見れば、説明なしに正確に理解・行動できるか？」
  → Yes：除外（例：申込書の枚数、記入欄の名称、画面上のボタン名）
  → No ：採用（例：保管先・提出先・理由・判断が必要な例外・担当窓口）

【callout_question の3条件チェック】
各Unitのcallout_questionが以下をすべて満たすか確認する：
  ① 知識ゼロでも答えを試みられる（日常感覚から考えられる）
  ② 学習後に「なるほど」という解決感が生まれる
  ③ 実物に書いていない判断・経験を問う

4分類別のcallout_question パターン（参考）：
  ①手順：「この手順を踏まないとどんなトラブルが起きると思いますか？」
  ②概念：「AとBは新人には同じものに見えます。何が違うか説明できますか？」
  ③判断軸：「この場面、自分で判断していい部分とそうでない部分はどこですか？」
  ④自己流：「マニュアル通りにやると実はうまくいかない場面はありますか？」

❌ NG例：「複数台の車をまとめて契約する場合、申込書はどう変わると思いますか？」← 知識テスト
✅ OK例：「車が10台ある会社の契約を担当します。1台ずつ申込書を書いていくとどんな手間が発生しますか？その手間を省くにはどうすればいいと思いますか？」← 思考起動

【Unitの構成順序（必ず守ること）】
  1. term（専門用語がある場合のみ）
  2. callout_question（必須・1ブロック）
  3. heading2 + paragraph（メイン説明：明示知＋暗黙知を統合）
  4. tip（暗黙知・自己流の工夫）
  5. callout（注意点・NG例・例外処理）
  6. bullet_list（まとめ・3点以内）

【使用できるbodyブロックtype（14種類のみ）】
callout_question / heading2 / heading3 / paragraph / bullet_list / numbered_list /
term / tip / callout / table / quiz / toggle / divider / code
※ 未定義のtypeはparagraphにフォールバックされる（警告扱い）

【テスト形式（2種類は別物・混同しないこと）】

◆ category_test（確認テスト）
  choices は { body, is_correct } のオブジェクト配列
  is_correct: true がちょうど1つ
  passing_score は 0〜100 の整数

◆ mastery_test（習得テスト）
  choices は文字列配列（オブジェクトではない）
  correct_index は正解の0始まりインデックス（0〜3）
  passing_rate は 0.0〜1.0 の小数（passing_score ではない）
  explanation は必須・titleフィールドは不要

【変動情報の扱い】
数値・料率・手数料等が登場する場合、該当箇所の直後に以下を設置する：
  🔄 変動情報：〇〇は定期的に変わります。最新情報は〇〇を確認してください。

## ■ JSON出力前チェックリスト（全項目確認してから出力すること）

- [ ] version: "1.0" が含まれているか
- [ ] title・description・categories が course: {} の中にあるか（トップレベルに直接置くと422エラー）
- [ ] reference_pages に「📋 この研修について」が含まれているか
- [ ] 各Unitのbody先頭に callout_question が1ブロックあるか
- [ ] callout_question は思考起動型になっているか（知識テスト型になっていないか）
- [ ] tip ブロックに ③判断軸 の暗黙知が含まれているか
- [ ] body内のtypeが上記14種類のみか
- [ ] 専門用語の term ブロックが本文より前に配置されているか
- [ ] 記載除外フィルターを適用したか
- [ ] category_test：choices が { body, is_correct } 配列か、is_correct: true がちょうど1つか
- [ ] mastery_test：choices が文字列配列か、correct_index が 0〜3 か、explanation が全問あるか
- [ ] category_test または mastery_test が各カテゴリに含まれているか
- [ ] OCRによる誤字・中国語漢字混入がないか（AI生成造語・意味取り違えにも注意）
- [ ] 全体のJSON文字列が1MB以内か（超える場合はコース分割）

## ■ JSON出力

\`\`\`json
{
  "version": "1.0",
  "course": {
    "title": "コース名",
    "description": "説明文（省略可）",
    "reference_pages": [...],
    "categories": [...]
  }
}
\`\`\`

JSONが大きい場合はカテゴリ単位で分割して出力し、
すべて出力し終えたら必ず「✅ 全カテゴリ出力完了」と明示すること。
```

## 補足：状況別の追加指示

| 状況 | ②再起動プロンプトの冒頭に追加する一文 |
|---|---|
| 回答が非常に薄い | 「回答が薄い設問は、添付資料から推測できる範囲で補完し、tipブロックに「要確認（SMEに再確認）」と注記してください。」 |
| 口頭起こしが長く脱線が多い | 「以下の文字起こしから、業務に関係する発言のみを抽出して処理してください。」 |
| JSONが長くなる場合 | 「今回はカテゴリ0〜2のみJSON出力してください。」（範囲を指定して分割投入） |
| 特定カテゴリだけ再生成したい | 「カテゴリXのJSONのみ再生成してください。暗黙知整理は以下を参照：（貼り付け）」 |
